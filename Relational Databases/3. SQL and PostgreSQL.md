## What Are Relational Databases, and How Do They Differ from Non-Relational Databases?
A *relation database* is a collection of data organized into tables. These tables are made of rows and columns.

Rows represent individual records: for example, if you're storing user's data, each row could represent a specific user. Columns represent the attributes or fields that describe each record. The table might include columns for common user attributes, such as name and national ID number.

One of the key characteristics of relational databases is that they allow us to connect different pieces of information by linking tables through common attributes, making it easier to see how they relate to each other.

Relational databases require a schema, which defines the overall structure of the database, including its tables, columns, data types, relationships and constraints.

All tables in a relational database have a **primary key**, which is a unique identifier for each row. They enable relationships between tables via **foreign keys**.

A query is a request for specific data from the database, that can be customized to fit specific needs.

Relational databases have a wide range of applications across industries such as healthcare, construction, business, gaming, education, government, e-commerce, social media and more. They offer several advantages, including scalability. They can handle large datasets, so they're perfect for complex, real-world applications. They also enforce data integrity through primary keys, foreign keys, and data types, which ensures that the data will be consistent and accurate.

In addition to relational databases, you can work with non-relational databases. The main difference between relational and non-relational databases is how the data is stored. Non-relational databases, also known as NoSQL databases, are more flexible. They store data in individual files that are not connected. Their data model is more flexible too.

While some non-relational databases may have a basic schema or data model, it is often less rigid than the schema defined in relational databases. You can add, modify, or remove data fields if necessary.

The decision to use a relational or non-relational database depends on various factors, including the nature of the data and the specific requirements of the application. You should evaluate the tradeoffs between these two approaches to choose the best one for your application.
